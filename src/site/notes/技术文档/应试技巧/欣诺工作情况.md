# DCI网元项目
主要用C语言编写 svn管理, 来公司的时候这个项目已经很成熟了,   主要负责电层卡的业务逻辑, 这个电层卡的主要作用就是用来进行光电转换来传输业务数据的, 我这边负责一些老产品的Bug修复, 新卡的功能适配, 比如告警, 性能, 板卡模式等, 

碰到的问题: 
业务问题
- 比如我碰到过一个告警误报的问题, 实际是模块误报的, 但我是怎么分析的呢, 我会先在公司搭一个环境去复现问题, 然后从最上层开始分析, 然后一层一层往下递进, 最终定位到最底层和模块通信, 获取寄存器值的位置
环境调试问题
- 比如我之前搭测试环境的时候拿到了一块坏卡, 应该怎么样让它能正常运行, 首先你就要看这块卡烧录了bsp没有, 然后再通过串口看内核启动日志是否正常, 正常的话, 就可以通过直连网口来传输最新版本的软件代码了, 然后重新启动设备
- 解决这个问题让我对整个软件项目的启动部署流程和升级过程都有了非常深刻的理解
# SONIC整体盒式项目
这个其实是我被分配到其他项目组和另外一个BSP同事一起协作完成的, 他负责驱动以及和客户对接部分, 我这边主要负责搭建程序框架和他联调业务功能和板卡功能

架构问题
就是关于这个问题呢, 我是在开始之前我会先和同事讨论总结有哪些功能, 这些功能之间有什么联系, 层级关系是什么, 从最开始就把框架定好, 这里由于产品种类也不止一个, 还做了驱动和业务逻辑解耦的分层, 把设备文件的相关信息做成了配置文件的形式, 后面使用别的产品也能保证代码的可重用性, 最大程度上节省了开发成本, 提高了代码质量

# DCI盒式机框板卡SONIC项目

这个项目就是基于Linux系统的二次开发, 北向接口使用go语言开发和南向接口使用python开发, 再往底层一点的核心程序都是用c/c++开发, 数据库选型使用redis数据库, 单卡之间的进程通信一般使用redis的发布订阅机制, 像Linux卡之间就使用的thrift这种rpc方案 , Linux卡和单片机使用的udp通信, 

更多Linux相关和比较前沿的软件技术
内存泄漏问题
源于客户反应连接器器升级之后没有恢复连接, 检查代码流程, 确认是存在网络断开连接恢复的流程, 然后复现问题, 制造
	
先用go的内存泄漏检测工具做一个简单的扫描, 然后再做一些操作时, 检查程序的内存消耗是否异常, 比如线程的分配啊, 栈帧的大小啊, 这类资源是否存在持续增长的情况, 再从代码出发结合日志
