# Docker 部署
## docker-compose.yml
### 字段解析
- `image`: 
- `external_url 'http://gitlab.example.com:8929'`：这里的8929是GitLab期望外部用户访问的端口。在容器内部，GitLab的Nginx会监听8929端口（这是容器内的端口）。为了让外部用户能通过宿主机IP或域名访问到这个服务，我们需要将宿主机的某个端口映射到容器的8929端口。例如，`ports: - "8929:8929"`。这样，外部用户访问`http://宿主机IP:8929`，流量会被Docker转发到容器的8929端口。如果映射的宿主机端口与容器内端口不一致（例如映射`8888:80`），那么外部用户需要访问`8888`，但GitLab生成的URL会是`http://gitlab.example.com:8929`，导致URL与实际访问端口不匹配（除非通过反向代理处理）。因此，通常建议让`external_url`中的端口与宿主机暴露的端口一致。
- `gitlab_rails['gitlab_shell_ssh_port'] = 2424`：这个配置告诉GitLab在生成SSH克隆URL时使用2424端口。但实际SSH服务在容器内监听的是22端口（GitLab默认）。为了让外部用户通过SSH访问容器内的SSH服务，我们需要将宿主机的某个端口映射到容器的22端口。例如，`ports: - "2424:22"`。这样，外部用户使用SSH命令`git clone ssh://git@宿主机IP:2424/...`时，连接会转发到容器的22端口。而GitLab会根据这个配置在项目页面上显示正确的SSH克隆地址（如`ssh://git@gitlab.example.com:2424/...`）。如果不设置，GitLab会默认使用22端口，如果宿主机映射的端口不是22，那么显示的克隆URL就不正确。
