# 网站错误码
## HTTP **422 Unprocessable Entity**
### 常见原因与排查思路

#### 1. CSRF Token 验证失败

- 你在浏览器中访问 GitLab 页面，但提交表单时（如登录、修改设置），GitLab 会校验 CSRF token。
    
- 如果 `external_url` 配置的域名/协议与当前请求的 `Host`/`X-Forwarded-Proto` 不匹配，Rails 会认为请求跨域，拒绝处理。
    
- 典型场景：
    
    - 通过 `http://192.168.0.200:8929` 访问页面，但表单提交时目标地址是 `https://oalh.dpdns.org:443`，导致 token 验证失败。
        
    - 或者代理头 `X-Forwarded-Proto` 与 `external_url` 的协议不一致，导致 Rails 无法正确构造表单地址。
        

#### 2. `Host` 头不匹配

- GitLab 要求请求的 `Host` 必须与 `external_url` 中的域名一致（默认情况下）。如果你用 IP 访问，而 `external_url` 是域名，可能被拒绝（但通常只会 404，不一定是 422）。
    
- 更隐蔽的是，当 Cloudflare 转发时，可能改写 `Host` 头，你需要确保 `proxy_set_header Host` 正确传递原始域名。
    

#### 3. `trusted_proxies` 未正确设置

- 如果你使用了 `X-Forwarded-Proto` 和 `X-Forwarded-For`，但 GitLab 不信任这些代理 IP，则 Rails 会忽略这些头，导致协议判断错误，进而可能触发安全限制。
    

#### 4. `force_ssl` 与 `external_url` 冲突

- 虽然你设置了 `force_ssl = false`，但如果 `external_url` 是 `https://`，Rails 仍可能在某些内部逻辑中期望请求是 HTTPS，与实际情况不符。
