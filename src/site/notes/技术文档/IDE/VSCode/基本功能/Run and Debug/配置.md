# `launch.json`
## 基本框架
```json
{
	"configurations": {
		# this write下述参数
	},
	"version": "..."
}
```
## 参数
- `"name": ""`
	- 配置名称, 自定义
- `"type": ""`
	- 调试类型, 不可乱填
		- `cppgdb`
		- `...`
- `"program": ""`
	- 开发宿主机上编译好的可执行程序完整路径, 不可乱填
	- 可用内置变量, 参考[[技术文档/IDE/VSCode/常用设置/通用配置语法#workspaceFolder\|通用配置语法#workspaceFolder]]
- `"args": ["", ""]`
	- 传递给可执行程序的命令行参数
- `"stopAtEntry": true/false`
	- 调试程序是否在 `main`处停止
- `"svdPath": ""`: 调试单片机时, 通过指定的`svd`文件能够列出该芯片的外设地址和字段信息, 调试视图依赖`Cortex-Debug`插件, 参考[[Cortex-Debug插件\|Cortex-Debug插件]]
- `"cwd": ""`
	- 用于查找依赖项和其他文件的当前工作目录
	- 即可执行程序中使用相对路径时, 相对路径的根路径就是这里指定的路径
	- 远程调试时, 填写开发宿主机上的路径, 需要满足以下要求
		- 和目标运行机上的可执行文件里符号表里源文件的相对路径的根路径要一致
			- **符号表里源文件的相对路径**参考`gdb`交互命令: [[技术文档/Linux/linux常用操作/编译工具链相关/工具链/调试器的使用/GDBClient#^67adbe\|info source]]
			- `eg. symbol table: ./src/libappd_cmd.cc, cwd: .../libappd`
				- 源文件完整路径: `.../libappd/src/libappd_cmd.cc`
- `"MIMode": "gdb"`
	- 调试器类型, 不可乱填
		- `gdb`
		- `...`
- `"miDebuggerPath": ""`
	- 如果要远程调试, 这里填入调试环境中用来连接调试服务器的客户端程序完整路径, 比如 `gdb`
	- 而且这个调试客户端程序的target侧架构(即解析可执行程序所使用的架构)必须和实际运行环境上 `gdbserver`的架构一致
- `"miDebuggerArgs": ""`
	- 指定客户端调试器的参数
- `"targetArchitecture": ""`
	- 指定调试器客户端解析的可执行程序运行架构, 不可乱填
		- `arm64`
		- `arm`
		- `amd64`
		- `...`
- `"miDebuggerServerAddress": "192.168.20.111:9988"`
	- 实际执行环境上gdbserver监听的IP地址和端口号
- `"setupCommands": [{}, ...]`
	- `"description": "..."`
		- 这个命令的描述
	- `"text": "..."`
		- 要发给调试器的命令
			- 参考[[技术文档/Linux/linux常用操作/编译工具链相关/工具链/调试器的使用/GDBClient#交互子命令\|GDBClient#交互子命令]]
		- 注意
			- 如果填写了参数`miDebuggerServerAddress`, 则不需要在这里添加以下命令
				- `target remote <gdbserver-ip>:<port>`
			- 如果填写了参数`targetArchitecture`, 则不需要在这里添加以下命令
				- `set architecture <ARCH>`
	- `"ignoreFailures": <true/false>`
		- 忽略命令执行出错, 继续执行命令
- `"sourceFileMap": {"<source-path>": "<target-path>", ...}`
	- 用于设置开发宿主机源码路径, 来匹配目标运行机上的可执行程序中符号表中解析到的源码路径, 以便IDE正确跳转到源代码处

# 常见问题
## 命中断点, 但是找不到源文件位置
- 通常是`cwd`属性没有设置, 或者错误设置
### 解决方案
- 设置`cwd`为正确的路径
