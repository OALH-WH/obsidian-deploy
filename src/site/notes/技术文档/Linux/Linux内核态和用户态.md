Linux 系统中，**用户态**和**内核态**是两种不同的 CPU 特权级别，用于隔离操作系统内核与用户程序，保障系统的稳定性和安全性。下面从概念、区别和切换方式三个方面进行说明。

---

### 一、用户态与内核态的概念

- **内核态（Kernel Mode）**  
    也称为管态、核心态。CPU 在此状态下可以执行**特权指令**（如中断管理、内存映射、I/O 操作），访问所有内存空间（包括内核空间和用户空间），拥有最高权限。操作系统内核运行在内核态。
    
- **用户态（User Mode）**  
    CPU 在此状态下只能执行**非特权指令**，访问受限的内存区域（用户空间），无法直接访问硬件或内核数据结构。用户进程（如 Shell、应用程序）运行在用户态。
    

---

### 二、两者的主要区别

|区别维度|用户态|内核态|
|---|---|---|
|**特权级别**|低（通常为 Ring 3）|高（通常为 Ring 0）|
|**访问能力**|只能访问用户空间，禁止 I/O 操作|可访问所有内存，直接操作硬件|
|**指令集**|只能使用非特权指令（如算术、逻辑运算）|可使用特权指令（如中断开关、内存管理指令）|
|**崩溃影响**|用户程序崩溃不会影响整个系统|内核崩溃会导致系统宕机|
|**系统资源**|通过系统调用请求内核服务|直接管理 CPU、内存、设备等资源|
|**切换开销**|进入内核态需要上下文切换，有一定开销|在内核态内切换上下文（如进程切换）开销较大|

---

### 三、用户态与内核态的切换

用户态程序需要执行特权操作（如读写文件、创建进程、网络通信）时，必须通过**系统调用**陷入内核态，由内核代为完成。此外，**中断**和**异常**也会导致从用户态切换到内核态。

#### 1. 系统调用（主动切换）

用户程序通过调用库函数（如 glibc 封装的 `read`、`write`）触发系统调用。具体流程：

- 用户程序将系统调用号和参数放入寄存器，执行 **`int 0x80`**（x86 传统方式）或 **`syscall`**（x86-64 专用指令）等陷入指令。
    
- CPU 硬件自动完成：
    
    - 切换到内核栈（每个进程有独立的内核栈）。
        
    - 保存用户态上下文（如 PC、寄存器、栈指针）到内核栈。
        
    - 将 CPU 特权级别提升为内核态。
        
    - 跳转到内核预设的系统调用入口（如 `entry_SYSCALL_64`）。
        
- 内核根据系统调用号执行对应内核函数，结果返回给用户态。
    
- 恢复上下文，通过 `iret` 或 `sysret` 指令返回用户态。
    

#### 2. 中断（被动切换）

外部设备（如磁盘、网卡）产生中断信号时，CPU 暂停当前执行的用户程序，强制切换到内核态执行中断处理程序。流程与系统调用类似，但中断入口是中断向量表指定的地址。

#### 3. 异常（被动切换）

用户程序执行非法操作（如除零、缺页异常）时，CPU 触发异常，切换到内核态进行相应处理（如发送信号或处理缺页）。

---

### 四、切换的关键数据结构

- **任务状态段（TSS）**：x86 架构中用于存储各特权级别的栈指针，当从用户态切换到内核态时，CPU 从 TSS 中加载内核栈（SS0、ESP0）。
    
- **内核栈**：每个进程在创建时会被分配一个内核栈，用于在内核态执行时保存上下文和局部变量。
