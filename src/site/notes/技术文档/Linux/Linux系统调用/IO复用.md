## 1. select

### 1.1 基本原理

select 是最早的 I/O 复用函数（POSIX 标准），它通过三个 `fd_set` 位图集合分别传递可读、可写、异常事件的文件描述符。用户调用 select 时，将这三个集合复制到内核，内核轮询检查集合中的每个描述符是否就绪，然后修改集合（清空未就绪的描述符），返回后用户需遍历所有描述符检查哪些仍存在于集合中。

### 1.2 特点

- **跨平台性好**：几乎所有 Unix-like 系统都支持。
    
- **使用位图表示描述符**：`fd_set` 大小固定，由 `FD_SETSIZE` 宏定义（通常为 1024），因此能监视的描述符数量有限。
    
- **每次调用都需重新设置集合**：因为内核会修改传入的集合，所以下次调用前必须重新添加关心的描述符。
    
- **线性扫描所有描述符**：无论描述符是否活跃，内核都要遍历全部集合，时间复杂度 O(n)。
    
- **支持三种事件类型**：读、写、异常。
    

### 1.3 缺点

- 监视数量有限制（1024）。
    
- 每次调用都需从用户空间拷贝整个集合到内核空间，开销随描述符数增加。
    
- 返回后用户需遍历所有描述符，效率低。
    
- 无法动态增加描述符，需重新创建集合。

## 2. poll

### 2.1 基本原理

poll 使用一个 `pollfd` 结构数组来监视文件描述符，每个结构包含描述符、关注的事件和返回的事件。调用时，用户将数组传给内核，内核填充每个结构的 `revents` 字段（就绪事件），然后返回。用户遍历数组检查 `revents`。

### 2.2 特点

- **无上限限制**：理论上可监视任意数量的描述符，仅受系统资源限制。
    
- **使用结构数组**，每个元素独立指定事件和接收结果，无需重新构造整个数组（但内核仍会修改 revents）。
    
- **事件类型更丰富**：支持更多类型（如优先级、错误等）。
    
- **仍采用线性扫描**：内核轮询数组中的每个描述符，时间复杂度 O(n)。
    
- **仍需拷贝数组**：每次调用都将整个数组从用户空间复制到内核。
    

### 2.3 缺点

- 当监视大量描述符时，每次调用都要复制整个数组，且内核线性扫描，性能随描述符数线性下降。
    
- 返回后仍需遍历整个数组，不能直接定位就绪的描述符。

## 3. epoll

### 3.1 基本原理

epoll 是 Linux 特有的高效 I/O 复用机制，由三个系统调用组成：`epoll_create` 创建 epoll 实例；`epoll_ctl` 添加/修改/删除要监视的描述符；`epoll_wait` 等待事件就绪。epoll 在内核中维护一个事件表（红黑树）和一个就绪链表。

- **事件注册**：通过 `epoll_ctl` 将描述符和关注事件注册到内核，只需注册一次，后续无需重复传入。
    
- **就绪通知**：当描述符就绪时，内核通过回调机制将其添加到就绪链表。
    
- **等待获取**：`epoll_wait` 只从就绪链表复制已就绪的描述符到用户空间，无需遍历全部。
    

### 3.2 特点

- **无监视数量限制**，仅受系统资源限制。
    
- **无需每次都拷贝全部描述符**：描述符注册后常驻内核，`epoll_wait` 只返回就绪事件，减少数据拷贝量。
    
- **时间复杂度 O(1)**：仅处理就绪的描述符，与监视总数无关。
    
- **支持两种触发模式**：
    
    - **水平触发（LT）**：默认模式，如果描述符就绪，`epoll_wait` 会一直返回该事件，直到被处理（类似 poll/select）。
        
    - **边缘触发（ET）**：仅当状态变化时通知一次，需要用户一次性读完数据（配合非阻塞 I/O 使用），效率更高，但编程复杂。
        
- **回调机制**：内核使用回调函数将就绪描述符加入就绪队列，避免轮询。
    

### 3.3 优点

- 高性能，尤其适合大量并发连接（如 C10K 问题）。
    
- 活跃连接数少时，epoll 仍比 poll/select 高效，因为无需遍历全部。
    
- 可动态修改监视事件，灵活性高。
    

### 3.4 缺点

- 仅 Linux 支持，不可移植。
    
- 边缘触发模式容易引发 bug，需谨慎使用。
    
- 对于少量描述符且活动频繁的场景，epoll 可能因回调机制产生轻微额外开销，但通常可忽略。

## 4. 三者对比总结

|特性|select|poll|epoll|
|---|---|---|---|
|**底层数据结构**|位图（fd_set）|结构数组（pollfd）|红黑树 + 就绪链表|
|**最大连接数限制**|有限（通常 1024）|无限制（受系统资源限制）|无限制|
|**数据拷贝**|每次调用都要从用户拷贝到内核|每次调用都要从用户拷贝到内核|描述符注册时拷贝一次，epoll_wait 只返回就绪事件|
|**事件获取方式**|线性扫描所有描述符|线性扫描所有描述符|从就绪链表获取，仅返回就绪事件|
|**时间复杂度**|O(n)|O(n)|O(1)（仅处理活跃连接）|
|**触发模式**|水平触发|水平触发|支持水平触发（LT）和边缘触发（ET）|
|**可移植性**|几乎所有平台|几乎所有平台|仅 Linux|
|**内核开销**|每次调用都遍历全部描述符|每次调用都遍历全部描述符|通过回调避免轮询，仅处理就绪描述符|
|**修改监视事件**|需重新设置整个集合|修改对应结构的事件字段，但内核仍需遍历|通过 epoll_ctl 动态修改，无需重建|

## 5. 适用场景

- **select**：适用于跨平台且连接数较少（例如小于 1024）的场景，如简单的小型服务。
    
- **poll**：适用于无连接数限制但连接数不是非常大的场景（如几千个），且对移植性有要求。
    
- **epoll**：高性能服务器，尤其适合处理大量并发连接（如 Web 服务器、即时通讯服务器），在 Linux 平台首选。
