# 背景
- Linux内核是"跨发行版"的, 内核文档必须同时适用于各种发行版和交叉编译
	- 所需安装的软件包工具依赖列表提供者
		- `Buildroot`
		- `Debian`
		- `Yocto`
		- `Ubuntu`
		- `Arch`
		- `...`
- 而编译内核依赖的软件包工具是发行版私有的, 因此并不由内核开发者维护, 而是发行版内核维护者
	- 发行版内核和官方内核的区别: 发行版内核是官方内核的一个工程实例, 即可拿来即用的Linux内核
	- 因此用发行版内核编译所需依赖用来编译你自己的Linux内核完全没问题(**所需软件包列表参考你开发环境的发行版内核类型**)
- 内核开发者们只提供依赖的工具最小版本
- 实际编译依赖的软件包工具清单由发行版内核维护者们提供, 即需要查看发行版内核文档
- Linux 内核的编译流程经历了：
	1. 在非 Linux 系统（Minix）上，使用 GNU 工具链编译最初的 Linux 内核
	2. 在能够运行 Linux 内核和 GNU 工具链的系统上，实现 Linux 内核的自举编译
	3. 随着内核和构建系统的发展，逐步形成并内置了一套 Host 构建工具（Kbuild / Kconfig）
	4. 在运行 Linux 的主机系统上，首次编译并发布了 Linux 发行版内核
	5. 在保持 Host 构建模型不变的前提下，引入交叉编译以支持多架构和嵌入式系统
- Linux内核的构建分为两部分
	- Host侧: 构建依赖, 默认已经支持, 只依赖于开发编译平台
		- 
	- Target侧: 构建目标, 由编译工具链的不同决定构建不同平台的目标
		- vmLinux
		- modules
		- dtb
- Linux内核和BSD的区别(Yocto就是一个更贴切BSD设计理念的构建系统, 即相比较于传统的Linux内核构建, 它还自己构建Host工具)
	- Linux内核只构建Target, 且默认Host侧是不可讨论的前提
		- vmLinux
		- modules
		- dtb
	- BSD
		- 除了Linux传统构建的Target, 还会构建自己的Host
		- 好处是什么?
	- 总结
		- 在BSP构建的系统上编译自己出问题一定是工程问题, 而不是运气问题, 因为传统Linux内核的编译, Host依赖是不可控的, 而BSP构建出来的系统自带可控可构建自己的最小Host工具链

# 编译配置 (保证能编译过, 并且为你想要的架构)
## `makefile`变量
- `ARCH`: 架构类型, 例如arm
	- 赋值类型为条件赋值, 参考[[make语法#4.2 `?=`（条件赋值）]]
- `CROSS_COMPILE`: 交叉编译器工具链前缀, 例如arm-none-linux-gnueabihf-
	- 赋值类型为条件赋值
## `makefile`目标
编译目标如`uImage, dtbs`这种目标, 需要 `LINUX-SOURCE-DIRECTORY/.config`的存在, 可通过手动配置`Kconfig`或者加载默认配置如`make XXX_defconfig`来填充`.config`
- `dtbs`: 编译设备树
- `uImage`: 编译 `uImage`格式的Linux内核
- `distclean`: 清理编译产生的全部内容, 一般比 `clean`清理的干净
- `XXX_defconfig`: 使用不同平台的默认配置, 参考[[结构简述#`configs` 保存不同平台的默认配置文件, 例如xxx_defconfig]]

## 实例
### `oprofile`软件包已经不被支持
- 解决方法
	- 内核的 `CONFIG_OPROFILE`**应该**可以设置为 `n`
### `mcelog`软件包已经不被支持
- 解决方法
	- 内核的 `CONFIG_MCELOG`**应该**可以设置为 `n`
