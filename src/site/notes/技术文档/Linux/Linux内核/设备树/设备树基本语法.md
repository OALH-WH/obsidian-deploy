# 设备树文件合并规则
- 设备树源文件（`.dts`）可以 `#include` 多个 `.dtsi` 文件，最终被编译成一个完整的设备树（`.dtb`）。根节点 `/` 中的属性会在所有文件中合并，如果同一个属性在多个地方定义，**后出现的会覆盖先出现的**。
- 一个`.dtb`文件只能由一个`.dts`文件生成, 当一个`.dtsi`文件引用另一个`.dtsi`文件的节点定义, 必须包含(`inlude`)它.
- 因此，如果你在 `myboard.dts` 中没有写 `#address-cells` 和 `#size-cells`，但包含的某个 `.dtsi` 中根节点定义了它们，那么最终根节点中**会有**这两个属性（取最后一次赋值的值）。如果没有在任何地方定义，则根节点中缺失这两个属性。
# /根节点
- 每个设备树文件只能有一个/根节点
- 不同的设备树文件之间可以存在多个/根节点, 它们之间不会发生冲突, 因为编译的时候这两个/根节点的内容会合并到一起
## 根节点下的compatible
- 字符串第一个值表示所使用的硬件设备名称
- 字符串第二个值表示所使用的`SOC`
- `Linux`内核会通过根节点的`compatible`, 来查看是否支持此设备, 如果支持设备才会启动`Linux`内核
- 如何匹配? 参考[[技术文档/Linux/Linux内核/设备树/设备匹配\|设备匹配]]

# 设备节点
## 命名格式
- `[node-label:]node-name[@unit-address]`
	- `node-name`: 节点名称, 应该能够清晰的描述节点的功能, 比如`uart1`, 表示这个节点是个`uart`外设
	- `unit-address`: 一般表示设备的地址`or`寄存器首地址, 如果节点没有的话可以不要, 比如`cpu@0`,`sram@10000000`, `soc`
	- `node-label`: 用于方便的访问节点, 而不需要输入完整的名称, 比如访问`cpu0:cpu@0`, 可以这样写`&cpu0 {}`
		- 引用之后, 就可以修改这个节点的属性/添加子节点
		- 引用节点覆写属性时, 应不被任何节点嵌套, 即在根节点外面
## 标准属性
### 数据类型
#### Bool类型
- 属性存在即为`True`, 否则为`False`
#### 字符串
- `"..."`
#### 无符号整数
- `<...>`
## 无符号整数列表
- `<...>, <...>`
#### 二进制数据
- `[...]`
#### 字符串列表
- `"...", "..."`
### compatible
- 兼容性属性, 是一个字符串列表, 用于将设备和驱动绑定
- 可以选择多个, 这个设备第一个兼容的厂商和驱动找不到, 就找第二个, 以此类推
- 为什么是由厂商和驱动组成?
	- `driver -> device tree -> device`
	- 因为设备树需要同时匹配设备和驱动
		- 比如一个总线设备热插拔事件, 会使驱动创建设备文件, 驱动创建设备文件需要找到设备树, 传递设备厂商和驱动模块
#### 格式
- `"[manufacturer1, ]model1"[, "manufacturer2, model2"...]`
	- `manufacturer`: 表示厂商
	- `model`: 一般是模块对应的驱动名称
### model
- 字符串, 用于描述开发板的名称或者设备模块信息, 比如名字什么的
### status
- 字符串, 状态信息
#### 格式
- `"STATUS"`
	- `"disabled"`
	- `"okay"`
	- `"error"`
### \#address-cells
- 用于拥有子节点的设备节点中, 来描述子节点的地址信息 
- 决定子节点`reg`属性中地址信息所占用的字长(32位, 即每个cell表示32位的数据)
#### 格式
- `<number>`
#### 示例
 ```devicetree
 / {
	 cpus {
		 #address-cells = <2>
		 #size-cells = <0>
		 cpu0: cpu@0 {
			 // 这里的第一个地址范围的起始地址由64位组成, 高32位是0x12, 低32位是0x14, 没有长度
			 reg = <0x12 0x14>
		 }
	 }
 }
 ```
### \#size-cells
- 用于拥有子节点的设备节点中, 来描述子节点的地址信息
- 决定子节点`reg`属性中长度信息所占用的字长(32位, 即每个cell表示32位的数据)
#### 格式
- `<number>`
#### 示例
 ```devicetree
 / {
	 cpus {
		 #address-cells = <1>
		 #size-cells = <1>
		 cpu0: cpu@0 {
			 // 这里的第一个地址范围的起始地址是32位的0x12, 长度是0x14(这里最大可描述的长度是2^32)
			 reg = <0x12 0x14>
		 }
	 }
 }
 ```
### reg
- 通过起始地址和长度, 描述地址范围
- 表示起始地址/长度的cell数量都可以通过[[技术文档/Linux/Linux内核/设备树/设备树基本语法#address-cells\|#address-cells]]和[[技术文档/Linux/Linux内核/设备树/设备树基本语法#size-cells\|#size-cells]]来指定
- **默认 `#address-cells`** 应为 **2**。
- **默认 `#size-cells`** 应为 **1**。
#### 格式
- `<address length[ address2 length2...]>`
	- `address`: 表示起始地址, 可由多个`cell`组成
	- `length`: 表示地址长度, 可由多个`cell`组成
### ranges
- 用于地址映射, 定义在父节点中, 表示和子节点的地址映射关系
- 可以为空, `eg. ranges;`, 说明子地址空间和父地址空间完全一样
	- 什么时候不一样? 比如总线下的设备, 总线设备的地址空间就和总线的地址不一样
#### 格式
- `<sub-node-start-addr parent-node-start-addr map-length>`
	- `sub-node-start-addr`: 映射表中, 子节点的起始地址
	- `parent-node-start-addr`: 映射表中, 父节点的起始地址
	- `length`: 映射长度
### name
- 记录节点名称, 已经弃用
### gpios
- 参考`Documentation/devicetree/bindings/gpio.txt`
- 表示使用的`GPIO pin`脚
#### 格式
- `[custom-]gpios = <&GPIOs NUM eletrical-level>`
	- `custom`: 自定义前缀
		- 比如`data-gpios`
	- `GPIOs`: 表示使用哪一类的`GPIO`
	- `NUM`: 具体到某一类的某一个`pin`脚
	- `eletrical-level`: 电平信号
		- `GPIO_ACTIVE_HIGH`: 宏值为0, 高电平有效
		- `GPIO_ACTIVE_LOW`:  宏值为1, 低电平有效
#### 示例
### gpio-controller
- 无值, 用来表示当前节点是一个`GPIO`控制器
### `#gpio-cells`
- 格式和`#address-cells`类似
- 值应该为`<2>`, 表示一共两个`cell`
	- 第一个表示`GPIO`编号, `eg. &gpioa 0`: 表示使用`PA0`
	- 第二个表示`GPIO`极性, `eg. GPIO_ACTIVE_HIGH`: 表示高电平有效
